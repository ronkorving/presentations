もともと，ソースコードの共有は当たり前だった
Once upon a time, sharing code used to be the norm The perceived value was in hardware, until Microsoft realized that software had huge potential as an industry. They were right.
The Open Source movement sought to protect the culture of sharing. The GNU Public License is one of the legal ways by which they have achieved this.
We owe a lot of the existence of Open Source today to friendly people like this guy.
And religious figures like this guy.
With all that Open Source around, being used by many of us on a daily basis, I find myself wondering why we don't contribute more. We owe so much to this movement, why aren't more people an active part of it?

オープンソースしていない理由

  • エンジニアのデバッグの方法
  • 社内のリーダー達のソフトウェアへの理解
The main reasons I've identified are the way we debug our code, and a lack of understanding software and engineering by the leadership of most of the companies we work at.

エンジニアのデバッグの方法

  • 私のバグではありません．そのライブラリーが壊れているんです
    （時にそれは正しいが，大抵は間違っている）
  • なんとなく済ませていませんか？
    1. Google / Stack Overflow!
    2. とりあえずの回避策→技術的負債に繋がる(T_T)
    3. ... 良い結果を産まない...
Let's start with how we debug our code. When we hit a wall, this is what tends to happen. Gut response: I didn't cause that bug, that library is broken (sometimes right, usually wrong) Our "solutions": - Google / Stack Overflow! - Workaround! (technical debt, hurray) - ... or worse

回避策がないなら...

「お客様，申し訳ございませんが
これは実現できません．不可能です．．．
専門家の私が言うので間違いありません」
"Dear Customer, this is impossible (trust me, I know tech!)" - "Not possible" or "Very expensive". - Like they know any better, right? - This is bad for business.

ソフトウェア開発者とは、
本来，問題解決者であることを意味します。

私たちが問題を解決しなければ、
他の誰が解決をしますか？

Being a software developer means being a problem solver first and foremost. Sadly, sometimes we forget this. If we don't solve the problem, who will?
The real solution is to get to the root cause of our problem. Stock photo guy knows what's up.

問題の根源に到達することは、
掘削していくことです

  1. 私のコードが問題なのか？
    (だいたいはこれが原因)
  2. 使っているライブラリーやフレームワークが
    問題なのか？（諦めちゃダメ〜）
  3. Node.jsの問題か，依存関係の問題か？
    （諦めちゃイヤ〜）
  4. JavaScriptのランタイムの問題か？
    （レアケースだけど0%ではないよね）

落ち着けば大丈夫

Getting to the root of the problem, is quite literally an exercise in digging. 1. Is the problem in my code? (most likely) 2. Is the problem in library/framework X? (debug it!) 3. (Is the problem in Node.js or its dependencies? debug it!) 4. Is the problem in the JavaScript runtime? (this actually happens) Don't panic. At our company, I really push for people to go through these steps one by one, and not give up until the root cause of the problem has been identified. This can be scary, but it is totally worth it.

バグを報告しよう！

  • バグを報告する責任があります
  • アナタは間違えているかもしれない
  • 間違っていると、ドキュメントの不備やAPIデザインの不具合が発見される可能性があります
Report that bug! - You have a responsibility to report bugs - You may be wrong, and your solution will become clear from this communication - Being wrong may identify poor documentation or bad API design Authors want to know about your problems so they can improve the library for you, themselves and others. Keep in mind that library authors can often have tunnel vision about the product they're working on. That can make your feedback invaluable.
でも納期は迫っている！
But my deadlines! I understand you would be worried about your deadlines. Reporting a bug tells you nothing about when you will be able to deliver your blocked feature.
自分で直して，プルリクエストを出そう．
So fix that bug and make a Pull Request! - "Fancy" developers are just people. - Have confidence in your own abilities. - Jump into the project with confidence, debug the problem, you will succeed. - Because existing contributors will often gladly help you when you get stuck (they will see a potential new contributor in you).

オープンソースのバグを積極的に修正すると(1/2)

  • 適切なレビューを受ける事が出来るため、修正したプログラムが確実に動作する
  • もし修正したプログラムがマージされなくても，あなたはそのプログラムを使うことは出来ます(だから納期には間に合う)
  • あなたにとって大切なソフトウェアのコミュニティメンバーになれます
Fixing bugs in Open Source software yourself... - Ensures the fix works, as it will receive proper review. - Will give you a public branch to continue to work from (deadline safe). - Become a member of a community around software in which you have a vested interest. - ...

オープンソースのバグを積極的に修正すると(2/2)

  • あなたのデバックスキルが向上します
  • そのソフトウェアのパフォーマンスの
    引き出し方に詳しくなれます
  • あなたがバクを修正することで
    世界中のエンジニアの役に立ちます
- Improves debugging of your own software. - Improves understanding of performance bottlenecks. - Prevents others from wasting time. That last point is especially important to me. As a community, if you can save a ton of people the effort of debugging the same issue, by spending 4 hours on a pull request yourself, you will save a lot of man-hours. I call that pretty productive. "Work smart, not hard", right?
When we acknowledge that our interests align, this should be the obvious course. Our bosses may not always recognise it, but in an Open Source world, all developers are co-workers. We're in this together.

問題の本質的な解決をするためには，仕事に自由が必要です

もしあなたのリーダーが，
エンジニアのオープンソースへ参加のメリットを理解していないのであれば，
あなたの仕事環境には問題があり，

Demand the freedom in your job to fix problems at the root. When your leaders fail to understand the real benefits of Open Source to your work, I believe your work environment has a serious flaw and you should demand better. Educate your leaders. Now if you're lucky and you work for a company where Open Source is already valued for what it is, awesome. But if that's not the case for you, inform your leaders and push for change.

公開することは本当にあなたの会社に
不利益を生みますか?そうじゃないなら,

  • みんなからの修正は0コストでしょ? いいね！

Open Source your own software Always ask yourself: Does sharing this really create a competitive disadvantage to us? If not, why not Open Source it? Contributions at zero cost? Yes please. It's cool! GitHub is a portfolio for yourself and your company.

The Wizcorp Pull Request challenge

  • GitとGitHubの基礎知識は必要
  • 応募者のプロジェクトでのコミュケーションのやり方を知ることができる
  • 応募者の問題解決能力を知ることができる
  • もし採用しなくても，応募者はその能力を残すことができる
  • 面接の時に意味のないプログラミングテストがいらない
At Wizcorp we have really made Open Source part of our DNA. And we even apply it to hiring. **Explain the PR challenge** - Requires applicants to know some Git and GitHub basics. - Shows us how applicants communicate with other people on a project. - Shows us an applicant's problem solving skills. - Allows applicants to show off their skills even when we don't hire. - No silly programming tests in an interview room.

コントリビュートしたくなったよね？

Super psyched to contribute yet? Sweet!

Some tips

  • 多くのプロジェクトにはコントリビュートの方法が書かれています．読みましょう！
  • 多くのプロジェクトには行動規範があります．それに従い，行動する事が大切です
  • たとえ他の人の行動がふさわしくないと思った時でも、あなたの寛大さは重要です
  • 相手のプロジェクトのスタイルを壊してはいけません
Some tips - A lot of projects document how to contribute (read it!). - Some projects advocate a Code of Conduct. Act professionally and respect it. It protects you and others working on the project. - Friendliness goes a long way, even when some others don't always seem to deserve it. - Stick to the style of the project you are contributing to. When you visit a friend’s house, you don’t start redecorating their home, right? Treat a project’s code the same, some people are spending their whole days in there.

コントリビューションしやすい場所

  • Documentation
  • Unit tests
  • Benchmarks
If you're a bit scared to get into a particular project and want to familiarize yourself with it a bit more first, these are 3 ways to add value to the project without having mastered it. Users are often better at writing documentation than the code authors themselves. And in the process, you will understand the code base more and more, and will be able to start contributing code. It is often also easy to start by writing tests and benchmarks for the project.
